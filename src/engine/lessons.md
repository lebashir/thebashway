# lessons

Distilled build pitfalls for lifeofbash. Each entry is one `- [tag] rule` line
(tag = an area, or `general`). The driver injects the relevant ones into each
basha's prompt as "Known pitfalls — do not repeat" and appends a new line when
a gate or a reviewing basha catches a real mistake (dedup automatic). Parsed by
`tools/orchestrator/lessons.ts`.

The file is split into two sections. `## Active` lessons are INJECTED into every
basha's prompt (filtered by area). `## Graduated` lessons are kept for history —
they're now encoded in code, automated by preflight, or moved into the
cold-review prompt template — and are NOT injected. The parser reads only the
`## Active` section.

Cross-project lessons graduate into the thebashway skill itself over time.

## Active

- [organs] Smoke must anchor on real signals (HTTP status + an expected marker), never grep rendered prose for words like "error"/"ENOENT" — substrate prose contains them legitimately (the smoke false-positive).
- [supabase] Before assuming a migration CREATES a table, check whether it already exists — `create table if not exists` silently no-ops on a pre-existing table (with possibly different columns/constraints). agent_events pre-existed (chat/cron rows, extra payload col, no source CHECK); confirm the real shape via information_schema before relying on the DDL you wrote.
- [integration] Re-verifying a multi-slice integration against a single surface's `--territory` will FAIL scope (it sees the other slice's files as overruns) — pass the union of all merged territories, or verify each slice against its own base. The gate chain (tsc/lint/test/build/smoke) is the real integrated-green signal.
- [launchd] Smoke a background job by running its EXACT plist ProgramArguments + EnvironmentVariables by hand before trusting the schedule (rails: don't ship a background job smoke can't exercise) — loading != running-correctly.
- [organs] Removing a placeholder export breaks its OWN test file too — when clearing a placeholder (e.g. placeholderAgentEvents), update both the source and its `*.test.ts`, and refresh the home snapshot (substrate edits like NOW.md make it stale → freshness fails).
- [a11y] Modal infrastructure goes a long way for the whole app — wiring `role="dialog"` + `aria-modal="true"` + `aria-labelledby` (via `React.useId()`) into the shared `Modal` component fixed every drawer in one edit. Confirm at the surface with a Playwright probe `document.querySelector('[role="dialog"][aria-modal="true"]')` after opening one — passes for all callers without per-caller changes.
- [organs] Lint rule `react-hooks/set-state-in-effect` forbids setState in a mount-only effect. Pattern: derive the initial state in `useState(() => ...)` from the source (e.g. searchParams), then a separate effect does only side-effects (e.g. `router.replace` to strip the consumed param). Don't fight the rule with `useEffect(() => setX(...), [])`.
- [organs] Introducing paired-row data (e.g. `transferPairId`) requires auditing every existing single-id mutation path (delete, update) — they're now wrong by default and will silently orphan a partner. (Wayfare Batch 3: the new transfer modal reused the old `deleteTransaction(id)` for delete; the partner row stayed in the DB with a dangling pair id.)
- [organs] URL-as-state with `inferPreset(from,to)` falls apart when a preset has no distinct from/to encoding (e.g. "All time" = "" / "" and "Custom range" starts as "" / ""). Track the ambiguous preset selection in local React state (a `customSelected` boolean) alongside the URL — derive from URL for unambiguous cases; let the local flag win for the ambiguous one. A `useEffect` resync trips `react-hooks/set-state-in-effect` and is the wrong shape; the boolean-flag pattern is clean.
- [spec] Spec territory must cover every type-cascade caller — adding a required field to a shared interface (e.g. `kind` on `AccountCardData`) forces every inline builder to update, and those callers fail scope-diff if they weren't declared. Walk the type's construction sites before declaring territory; or expand at integration time and document the gap. Bashas can also misread "scope ok" from a wider-territory retry as if the original spec passed — driver must re-verify with the declared territory before integrating.
- [preflight] preflight-seed.txt entries are repo-root-relative, NOT surface-cwd-relative (matches worktree-seed.ts which resolves against `repoRoot`). PreflightSurface carries an optional `repoRoot` field for this; cli.ts must pass `REPO_ROOT`. Symptom of the bug: preflight reports a seed missing that actually exists at the repo root (it was looking under `<surface.cwd>/<seed>`).
- [supabase] [infra] `supabase functions deploy` CLI fails on the work laptop with `invalid peer certificate: UnknownIssuer` when Deno fetches JSR (`jsr.io`) — and the failure persists across networks (it's not Tabby-network-bound; the corporate root cert intercepts host TLS even on personal wifi, but the supabase CLI's Docker container can't see the host keychain). DO NOT WASTE TIME debugging via off-network deploy or env vars (NODE_TLS_REJECT_UNAUTHORIZED, DENO_TLS_CA_STORE) — they don't propagate into the container. **Workaround that works:** deploy via the Supabase MCP `deploy_edge_function` tool instead. That's how `wf-capture` was deployed (see its file header comment), and how `lesson-capture` was deployed 2026-05-29 (in one MCP call with `files: [{name, content}]`). Pass `verify_jwt: false` for endpoints that use custom secret auth.
- [mcp] Newly-added MCP tools (e.g. via a new `tools/mcp/<group>-tools.ts` and entry in `tools/mcp/tools.ts`) require restarting the Claude Code session to be picked up — the running MCP server doesn't hot-reload. Symptom: ToolSearch returns "No matching deferred tools found" for tools that were just shipped and merged on main.

- [organs] Any new `app/api/*` route that must be reached by a machine (Vercel Cron, webhooks) needs an explicit bypass in `src/proxy.ts` (the Next 16 password gate — NOT `middleware.ts`). The matcher catches everything non-static, so without `if (pathname.startsWith("/api/cron/")) return NextResponse.next()` the route 307-redirects to `/login` and the caller (no session cookie) never runs it. The route must still self-authenticate (e.g. `CRON_SECRET` bearer). Smoke it against PROD (`curl` the deployed route) — local dev has `AUTH_SECRET` unset so the gate is inactive and hides the bug.

- [verify] Establish a GREEN baseline on main BEFORE dispatching bashas, or every basha reports the same pre-existing failures and you can't separate regression from baseline. The tools-surface gate chain is `[test, validate]`: a single missing `import { beforeEach }` ReferenceError makes the WHOLE `bun test` exit 1 (even with "0 fail"), and substrate `validate` only requires `name`/`relationship` on `people/` files (it wrongly flagged `self/` until scoped). Run both on main first; fix baseline reds (driver inline) and commit before the wave.
- [worktree] tools-surface worktrees need `node_modules` symlinked from the main checkout — they're pnpm-managed and NOT in `preflight-seed.txt`, so a fresh worktree can't resolve `@supabase/supabase-js` etc. After `git worktree add`: `ln -sfn <main>/node_modules <wt>/node_modules` and the same for `tools/` and `organs/`. (organs `pnpm build` may still reject the symlinked node_modules under Turbopack — run the integrated build on the main checkout, not in-worktree.)
- [queue] The queue-ops serializer (`claimNextN`/`markDone`) is LOSSY on round-trip — it dropped both HTML-comment context blocks AND a `Clarifications` line when it rewrote queue.md at claim time. Prefer MANUAL queue.md edits for status changes (`- [ ] @unclaimed` → `- [x] @done`); if you must use `claim`/`done`, `git diff` queue.md afterward and restore any dropped content.
- [hygiene] Driver must never `git commit -am` — it sweeps unrelated working-tree changes into your commit. This session it bundled a background wrap-up-audit's `memory/operating-lessons-trust.jsonl` append + the queue.md claim-state into an inbox-filer fix commit. Stage explicit paths only (`git add <path> ...`). (Generalizes the graduated preflight `git add -A` lesson to the driver's own commits.)
- [reviewer-can-be-wrong] A cold-reviewer finding can both be technically wrong AND conflict with the basha's tested design. This wave: the "skip calendar-sync/whatsapp-sync notes" suggestion broke an existing "genuine capture is not skipped" test (the basha deliberately lets the CoS triage those), and the reviewer's `update_event` datetime/timezone guard was itself buggy (timezone is shared across start/end, so it wrongly rejected updating only `start`). Re-run the unit tests after applying any reviewer fix, and implement the CORRECT guard, not the reviewer's verbatim one.
- [infra] The auto-mode classifier auto-DENIES `apply_migration` (and other shared-DB MCP writes) even under a standing "do wave X" authorization, and the driver cannot self-allowlist (the settings.local.json self-edit is also classifier-blocked). Get migrations applied via Bashir's explicit approval, his `!` prefix, or by handing him the SQL — don't assume the driver can apply a prod migration mid-loop. Commit the SQL file in the same change regardless (source-of-record). (Wave B: `tg_pending_actions` shipped as code + file but the apply was blocked → deferred to Bashir.)
- [rail] A "reach other people" guard must inspect the FOUND resource, not just the request text. `move`/`cancel "my 3pm standup"` names no attendees, but the looked-up calendar event has them — rescheduling/cancelling notifies those people, bypassing the human-only Approve rail. After an entity lookup (event/thread/contact), re-check its attendee/recipient set BEFORE mutating, and defer to the rail if it reaches anyone. (Wave B cos-calendar cold review.)
- [zone] The operating-lessons `source` string controls auto-apply: `"wrap-up-audit"` IS in `TRUSTED_SOURCES` (the trusted interactive-session path from CLAUDE.md), so it auto-applies safe-area lessons. An AUTONOMOUS `claude -p` job that must only ever PROPOSE needs a NON-trusted source (Wave B used `"wrap-up-job"`). Verify source membership against `TRUSTED_SOURCES` + the `canAutoApply`/`proposeLesson` branch, not the source's name.
- [cos] Telegram Approve-rail buttons MUST carry the `tg_pending_actions` row uuid in `callback_data` (`approve:<id>`/`cancel:<id>`), NOT an upstream id (Gmail message id, action kind, etc.) — the webhook (`organs/src/app/api/telegram/webhook/route.ts`) looks the row up by that id, so any other value makes Approve/Cancel a silent no-op. Capture the id the enqueue INSERT returns and thread it into the buttons. (Wave C: surfaced independently in BOTH email-triage and work-triage.)
- [cos] A CoS job that resolves `telegram_chat_id` (or any config/secret) from `loadIngestConfig()`/env must expose it as an INJECTABLE dep — else its tests pass only on a machine that has `~/.config/lifeofbash/ingest.json` and fail on a clean checkout/CI. Inject the chat id (and Supabase client) in every enqueue-path test. (Wave C, caught by cold review in relationship-upkeep + work-triage.)
- [cos] `runSendExecutor` must FILTER its SELECT to the kinds it can send (`.eq("kind","email")`). A 5-min launchd poller that selects an approved-but-unsendable kind (slack_message/jira_comment, no executor yet) re-selects + re-logs it every tick (busy-loop). Filter at the query, keep a defensive skip branch, and never mark such rows `failed` (poisons them before the future executor exists).
- [cos] Reuse `kind='email'` for any draft the existing Gmail executor can send (reach-outs, replies) instead of inventing a new pending-action kind — a new kind means a new executor branch (more work + a shared-file collision when parallel items both edit `cos-send-rail.ts`). Only invent a kind when the channel truly differs (Slack/Jira), then defer/flag the executor as follow-on.
- [organs] [worktree] A worktree needs a REAL `node_modules` to build — a symlink to the main checkout's node_modules passes tsc/vitest but **Turbopack rejects it** ("Symlink organs/node_modules is invalid, it points out of the filesystem root"). Use `bun run thebashway spawn-worktree` which runs `pnpm install --frozen-lockfile --prefer-offline` (~4s on a warm store, mostly hardlinks). Never symlink node_modules into a Next/Turbopack worktree.
- [meta] `bun run thebashway claim N` clamps N to >=1 (`Math.max(1, …)`), so `claim 0` does NOT mean "show/dry-run" — it CLAIMS one item. To inspect the queue without claiming, read `queue.md` or use `intake-list`; never `claim 0`.
- [meta] A cold-review basha has zero conversation context AND no harness tools (it can't see the Workflow tool, etc.), so it may flag a real harness feature as "invented" or re-raise something already settled in conversation (e.g. a freeze clarification). Verify its tooling-existence/context claims before acting — but its code-correctness findings are gold (it caught a vacuous test by disabling the fix).
- [organs] [meta] A green-gate-only fix (e.g. a money `'use server'` action with no unit test) still needs its rare unit test to ASSERT real state changes, not just `resolves.toBeUndefined()` — a vacuous test gives false safety (disabling the fix still passes it). If the harness already tracks the state (mock update arrays), assert on it.
- [meta] The real `audit <target>` and `drain` commands SPAWN `claude -p`; even `audit --dry-run` spawns the Opus finder/verify/shape fan-out (only the queue WRITE is skipped). Only `audit-plan` and `drain --dry-run` are spawn-free. NEVER put the real `audit`/`drain` in a smoke/test command — it launches real bashas and mutates the queue.
- [general] `appendLesson(path, lesson)` takes a `{ tag, rule }` Lesson OBJECT, not a string. The tools surface gate chain has NO tsc, so a default-deps type mismatch like this passes the green gate and only surfaces at runtime — run `bunx tsc --noEmit` BY HAND after touching orchestrator code (the cores are tested; the real default deps are not).
- [verify] A per-unit re-verify must run where the unit's commits LIVE (the worktree), not the primary checkout: `verify/index.ts` derives repoRoot from `import.meta.dir` and diffs `base..HEAD`, so invoking it from the primary checkout (whose HEAD lacks the worktree's commits) silently verifies an UNCHANGED tree — the gate passes without exercising the code. Run verify with `cwd` inside the worktree's `tools/` and read that worktree's manifest.
- [meta] `drain --surface S` only configures the build/verify CHAIN — the CLAIM must independently filter to that surface (every territory glob under the surface dir). A mixed-surface queue (organs + tools items) otherwise cross-claims: a `--surface organs` drain grabs the tools item and builds it with the organs chain. Single-surface test queues hide this; the first live multi-surface `drain --dry-run` exposed it. Filter claims by `inSurface(item, surfaceDir)`.
- [meta] `drain`'s default `integrateUnit` checks out the integration branch in the PRIMARY repo checkout (then restores), so two surface drains cannot run concurrently — they fight over the primary checkout's HEAD. v1 drain is strictly serial (one surface at a time); a concurrent design needs a dedicated integration worktree.
- [infra] The interactive `drain` process's OWN `notifyTelegram` fetch hits the Tabby self-signed cert (`SELF_SIGNED_CERT_IN_CHAIN`) and silently no-ops — `headlessEnv` sets `NODE_TLS_REJECT_UNAUTHORIZED=0` for the basha child and the tools surface config sets it for verify subprocesses, but NOT for the drain process itself. `cmdDrain` now sets it on the process env. (Caught by the first live drain: the run succeeded but the digest never reached Telegram.)
- [rail] A safety rail that lives only as SKILL/decisions PROSE (e.g. "park anything that reaches a person or destroys data") is enforced ONLY by the interactive human driver — there is NO equivalent code in `drain.ts`. Any feature that AUTOMATES the driver away (headless drain, the `design` door) must RE-ADD that rail as deterministic TESTED code that LLM output can never override (the design door's `classifyIrreversible`). When you remove a human checkpoint, first enumerate which rails that human was silently enforcing. (Design-door spec cold review, 2026-06-05.)
- [rail] A sensitivity deny-list keyed on a hand-curated FILE list is fail-OPEN: it silently drifts as new sensitive files land, so a neutral-worded edit to a NEW send/delete job slips every net and (in an autonomous loop) auto-deploys (a `calendar-command.ts` edit that DELETES events was the proven bypass). Prefer a DIRECTORY-level default-sensitive rule (the whole `tools/jobs/**` is person-reaching-by-default) so a new file is safe-by-default; exempt known-safe ones explicitly. For an irreversible/external action, err toward over-matching — a false @needs-intake costs one glance.
- [queue] A queue item's TITLE is its identity: `markDone`/`markBlocked`/`claim` resolve `items.find(i => i.title === t)` (FIRST match), `dependsOn` references a title, and a title-membership landing check keys on it. Duplicate titles in one decomposition collapse all three — one success satisfies two same-titled members and an atomic-landing guard ships a HALF-BUILT feature. A machine decomposition must validate unique titles (force dups to @needs-intake). (Design-door break-it, 2026-06-05.)
- [queue] `isClaimable` treats a `dependsOn` whose title isn't found as SATISFIED ("missing dep is a no-op"), and a dependent of a never-@done parent stays unclaimable + silent. Validate a machine-emitted dep graph against the batch (dangling title → hold the dependent; cycle → hold) and CASCADE a held parent to its dependents — never rely on `isClaimable` to enforce ordering it cannot see.
- [meta] An autonomous multi-task feature must drain ONLY its own items (an allow-list of this run's titles — `claimNextN`'s `allowTitles`; else a pre-existing @unclaimed item is built and folded into the feature's land) and land ONLY when EVERY build-ready member reached @done. `blocked===0` is NOT sufficient: a member behind a gated/unfinished dep is neither succeeded nor blocked (stranded). Check explicit per-member @done; a blocked/stranded member → stage, never deploy half a feature. (Design-door diff cold review, 2026-06-05.)
- [organs] Production smoke (`scripts/smoke-prod.ts`) GETs only REGISTERED routes (the organ registry + a hardcoded sub-route list) — a brand-NEW page/route passes by being IGNORED, not exercised. "smoke green" therefore does NOT prove a new route works. Never auto-deploy a feature that adds a new organs route until smoke discovers/covers it; build it but STAGE for review (what the `design` door does for organs UI).
- [infra] To run `tsc` by hand on the tools surface you must first add `@types/bun` (dev) — `tsconfig.json` has `types:["bun"]` but the package isn't in the workspace, so bare `bunx tsc --noEmit` fails `TS2688 "Cannot find type definition file for 'bun'"`. After adding it there is a ~300-error PRE-EXISTING backlog (the surface never had a tsc gate) — filter tsc output to your CHANGED files. Revert `package.json`/`bun.lock` afterward if you don't intend to fix the gate. (Extends the "run tsc by hand" lesson with the actual blocker.)
- [rail] The per-project brief (north star) is written ONLY by the two human-present cold-start entry points: `seedBriefIfAbsent` (the idempotent init draft — exists today) and `writeConfirmedBrief` (the brief-command interview write — the second human-present writer; until that command lands the human-present write is a hand-edit of brief.ts, itself a spec-sanctioned human path). `src/engine/brief.ts` exports NO writer, so the design/audit/drain/digest stages export AND import no brief writer; they can only `loadBrief` + evaluate. A milestone reflection that wants to grow the brief (incl. conventions/glossary) PROPOSES via emitPark/sinks — there is no engine code path that writeFileSyncs brief.ts. (Codifies INV-A as tested code, per the line-57 discipline that an autonomous-door rail must be code, not prose.)

## Graduated

These pitfalls were caught at least once but are now prevented elsewhere. Kept
for history; NOT injected into bashas.

- [preflight] The regen-commit step used `git status --porcelain` + `git add -A`, so any unrelated uncommitted work present at preflight time got swept into the "refresh derived artifacts" commit and pushed (2026-05-29: bundled docs/share-with-friend/ + a binary zip onto main). FIXED in `preflight.ts`: dirty-check + add are now scoped to `surface.derived` (repo-root-relative), run from repo root. Lesson for the pattern: never `git add -A` in automated commits — stage explicit declared paths only.

**Now encoded in code:**
- [general] Model per basha role: Opus for plan/think/design/review (reason properly), Sonnet for building (fast); never haiku (thrashes). Unmetered — pick for quality/speed, not cost → skill Bashas section.
- [general] Parsers that read list items must strip HTML comments first → done in `lessons.ts` / `queue.ts`.
- [organs] Run `pnpm build`, never `next build` (skips prebuild snapshot regen) → encoded in `config.ts` `chain.build`.

**Now automated by preflight (`bun run thebashway preflight`):**
- [worktree] Base worktrees on local HEAD when there are unpushed commits → preflight pushes first.
- [worktree] Push local commits before dispatching worktree bashas → preflight pushes first.
- [worktree] Fresh worktrees do NOT contain gitignored files (`organs/.env.local`) → worktree-seed copies from `preflight-seed.txt` on spawn.
- [organs] Before dispatching ANY worktree basha, refresh+commit+push the home snapshot → preflight runs `pnpm gen:home`, commits + pushes if diff.
- [organs] When a slice changes content that feeds a derived artifact (e.g. home-snapshot.json), include `organs/src/generated/**` in `--territory` → preflight regenerates; territory includes generated/** by default.

**Now automated at integrate-time:**
- [general] Bashas can leave file copies in the host checkout (cwd is main repo, not worktree) → integrate-time `git clean` the work path before merge.
- [organs] Deleting a Next.js App Router page leaves stale type validators in `.next/types/validator.ts` → integrate-time detects deleted routes in diff and `rm -rf .next` before build.

**Now in the cold-review prompt template (`cold-review-prompt.md`):**
- [general] A cold reviewer's suggested fix can be WRONG — judge it against actual behavior before applying.
- [reviewer-can-be-wrong] A cold reviewer that only sees the git diff can't see Supabase migrations applied via MCP — the driver now passes `list_migrations` output as reviewer input.
- [verification] Probes that grep for text must match the actual rendered text — case / substring traps lie.
- [verification] Test interactive shapes by what a user actually sees — render-as-`<div>` when no handler.
- [supabase] Reuse the user's existing board columns — check live `columns` before introducing a near-duplicate.

**Now in the skill body:**
- [general] Update NOW.md on EVERY state change → skill step 10 (Digest).
- [general] Orientation/machinery docs (CLAUDE.md, README, USAGE, docs/) are the BUILD zone → skill Project bindings section.
