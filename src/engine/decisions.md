# decision rules (Loop A)

The intake decision/preference store. Read before intake escalates a question to
Bashir; appended when a question is answered. Format mirrors `lessons.md`:
`- [tag] rule`. Tag `decision` = always-on global default; tag = an area
(e.g. `tools`, `organs`) = injected only for items in that area.

## Active

- [decision] Prefer the lean option; do not add config/flags/abstractions a task did not ask for.
- [decision] Anything that reaches a real person (email/message) or deletes unrecoverable data parks for Bashir, never auto-proceeds.
- [decision] Auto-promotion / auto-intake of machine-captured work is conservative by default: promote only what is high-confidence-resolvable from the decision store + the codebase; leave ambiguous items for the human with the open question recorded — do not guess-and-lean-on-the-downstream-gate.
- [tools] The capture sweep harvests wrap-up-audit candidates into the build queue, but ONLY engineering-flavored bullets (a concrete defect/work signal) — life/behavioral candidates stay on the operating-lessons path. Deterministic keyword filtering is coarse; the @needs-intake gate + LLM intake is the real triage.
- [decision] Headless/scheduled audit runs MUST force all findings to @needs-intake, OR claim must refuse origin:auto + @unclaimed items without an explicit --allow-auto-build flag. The build-ready relaxation (audit findings can be @unclaimed) is sound ONLY in the interactive case where a human reads the report and is the gate before any drain.
- [decision] Migration vs lean code guard: default to the lean guard (fail-safe, no schema change) over a DB migration unless the migration is explicitly chosen. (2026-06-05: unmarkCommitmentPaid — throw if txn missing, no amount column.)
- [decision] Don't build a new test harness for a small fix; follow the existing test convention and rely on the green gate. (2026-06-05: money 'use server' actions are not unit-tested — upsertRule markReal shipped on the gate, no new Supabase-mock harness.)
- [organs] Forecast/model math that treats an entity as a fixed planned amount: prefer the planned-amount reading over joining actual-transaction data, unless actual-spend precision is explicitly required. (2026-06-05: forecast monthToDateSpent = commitment planned amount, no txn join.)
- [organs] When correcting an underlying computation, prefer the fix that does NOT regress an existing edit/UI affordance. (2026-06-05: getGoalsData — filter archived only for progress, keep archived in the goal-edit dropdown.)
- [tools] Idempotency for a re-fetched tagged/unread email (e.g. the #lesson pre-classifier): reuse the scored-email path's PERSISTED on-conflict marker keyed by source_id — it survives both re-fetch AND proposal rejection.
